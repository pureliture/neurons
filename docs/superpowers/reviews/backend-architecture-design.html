Backend Architecture Design Document

rag-ingress-queue
백엔드 아키텍처 설계

컴포넌트 드리븐 레이어드 아키텍처를 기반으로 한 패키지 구조 설계. 기능 단위 패키징(Feature-based Packaging) + 포트-어댑터 패턴 적용.

포트-어댑터 패턴 개념 정리
헥사고날 아키텍처의 핵심 개념입니다. 포트(Port)는 우리 도메인이 정의한 인터페이스이고, 어댑터(Adapter)는 그 인터페이스를 외부 기술/서비스에 맞게 구현한 것입니다. 인프라와 어댑터는 같은 개념의 두 종류입니다.

Port = 인터페이스 / Adapter = 구현체

IngestPublisher (Port, 인터페이스) ← 우리 도메인이 "메시지를 보낸다"고 정의
NatsIngestPublisher (Adapter, 구현체) ← NATS JetStream으로 실제로 보냄

RagTargetAdapter (Port, 인터페이스) ← 우리 도메인이 "타겟에 전달한다"고 정의
RagFlowTargetAdapter (Adapter, 구현체) ← RAGFlow REST API로 실제로 전달

인프라 어댑터

우리 시스템이 사용하는 기술 인프라를 도메인에 맞게 감쌉니다.

예: NATS (메시징), Caffeine (캐시), WebClient (HTTP 클라이언트 설정)

→ "우리가 기술을 도메인 방식으로 사용한다"

외부 서비스 어댑터

외부 비즈니스 서비스를 우리 도메인에 맞게 감쌉니다.

예: RAGFlow (RAG 타겟), GitLab (소스 저장소), 외부 인증 서버

→ "외부 서비스를 우리 도메인 방식으로 사용한다"

참조 구조 — sidebeam-backend (컴포넌트 드리븐 레이어드)
sidebeam-backend는 기능(도메인) 단위로 패키지를 분리하고, 각 기능 내부에서 계층형 구조를 유지합니다. 컴포넌트 드리븐 + 레이어드의 혼합 패턴입니다.

제안 구조 — 컴포넌트 드리븐 레이어드 + 포트-어댑터
기능 단위 패키징 + 포트/어댑터 명확 분리를 적용한 제안 구조입니다. Port는 기능별로 존재하고, Adapter는 기술 종류(인프라/외부)에 따라 분리됩니다.

계층 의존성 규칙 (ArchUnit 검증 대상)
아래 규칙을 ArchUnit 테스트로 코드화하여 CI에서 자동 검증합니다. 위반 시 빌드가 실패하므로 아키텍처 드리프트를 방지합니다.
api (표현)
service (응용)
domain (도메인)
port (포트)
adapter (어댑터)
api → service → domain → port ← adapter
common (exception, logging, config)는 모든 계층에서 참조 가능
✗ service는 api를 알 수 없음 (순환 참조 방지)
✗ domain은 adapter를 알 수 없음 (순수성 유지)
✗ adapter 내부 구현은 다른 adapter에 노출 불가
세 구조 비교표
패턴/속성 현재 rag-ingress-queue sidebeam-backend 제안 구조
패키징 방식 기술 계층 기능 단위 기능 단위 + 포트-어댑터
포트-어댑터 분리 Port/Adapter 있음, 패키지 혼합 논리적 Port 없음 Port/Adapter 명확 분리
어댑터 분류 인프라/외부 구분 없음 external/로 분리 (Port 없음) adapter/infra vs adapter/ext
예외 처리 계층 없음, Controller 직접 조립 ApplicationException + ErrorCode + @RestControllerAdvice sidebeam 스타일 적용
설정 외부화 @Value 산재 @ConfigurationProperties + Jasypt @ConfigurationProperties 적용
아키텍처 테스트 없음 ArchUnit 3개 규칙 ArchUnit 5개 규칙 제안
백프레셔/압력 게이트 TargetPressure + RagFlowPressurePolicy 해당 없음 유지
로그 마스킹 Payload 검증만 Logback Converter (런타임) sidebeam 스타일 적용
Correlation ID / MDC 없음 필터 + MDC + 응답 반사 적용
커버리지 강제 없음 Jacoco 80/70 report-only → threshold 순차
기능(컴포넌트)별 상세 구조
각 기능은 독립적으로 이해할 수 있는 단위입니다. 기능 간 의존은 최소화하고, 공통 인프라(common)와 포트(port)를 통해서만 연결됩니다.
📥
ingest
  • POST /v1/ingest/enqueue 처리
  • 요청 검증 (validation + redaction)
  • JetStream publish (via queue.port)
  • Idempotency key 관리
  • api → service → domain → queue.port 의존
🚚
delivery
  • Worker pull consumer 루프
  • TargetProfile별 라우팅
  • Pressure check → deliver / skip
  • Retry / nak / ack 정책
  • worker → service → domain → target.port 의존
📡
status
  • GET /status 응답 생성
  • Queue 상태 (pending, inFlight, deadLetter)
  • Target 상태 (pressure, adapter snapshot)
  • RAGFlow 설정에 직접 의존하지 않음
  • api → service → target.port 의존
📨
queue (port)
  • IngestPublisher — 메시지 발행 인터페이스
  • IngestConsumer — 메시지 소비 인터페이스
  • 도메인이 큐 기술을 몰라도 됨
  • NATS, Kafka, RabbitMQ 교체 가능
  • 구현체는 adapter/infra/nats/
🎯
target (port)
  • RagTargetAdapter — 타겟 전달 인터페이스
  • checkPressure(), deliver(), snapshot()
  • 도메인이 RAGFlow를 몰라도 됨
  • Qdrant, Weaviate 등 교체 가능
  • 구현체는 adapter/ext/ragflow/
⚙️
adapter/infra
  • NatsIngestPublisher — Port 구현 (메시징)
  • NatsIngestConsumer — Port 구현 (메시징)
  • NatsJetStreamProvisioner — Stream 생성
  • 기술 인프라를 도메인 방식으로 감쌈
  • → "우리가 기술을 사용한다"
🔌
adapter/ext
  • RagFlowTargetAdapter — Port 구현 (RAG 타겟)
  • HttpRagFlowGateway — HTTP 클라이언트
  • RagFlowPressurePolicy — 압력 계산
  • 외부 서비스를 도메인 방식으로 감쌈
  • → "외부 서비스를 우리가 사용한다"
🛠️
common
  • config: Spring 설정, @ConfigurationProperties
  • exception: 계층형 예외 + ErrorCode enum
  • logging: SensitiveDataMaskingConverter
  • web: CorrelationIdFilter, ApiKeyAuthFilter
  • 모든 기능이 참조 가능 (최하위 계층)
