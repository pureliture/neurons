Audit Override

물리적 분산 컨테이너화 철회 및 Modular Monolith 권장

본 문서의 초기 초안은 4개의 기능을 물리적 네트워크 컨테이너로 분리할 것을 제안했으나, 아키텍처 감사(Architecture Audit) 결과 이는 오버엔지니어링(분산 트랜잭션의 저주, SPOF 연쇄 결합)으로 판정되었습니다.

  • 모듈식 모놀리스(Modular Monolith): 물리적 컨테이너로 쪼개지 말고, 단일 컨테이너 내부에서 인터페이스와 클래스 경계만 엄격히 나누는 방식을 채택합니다.
  • DB Adapter 설계 리스크: PostgreSQL과 SQLite 간의 SQL 방언 및 트랜잭션 격리(MVCC vs 직렬화) 차이를 런타임에 커스텀 변환하는 것은 위험합니다. 검증된 ORM(SQLAlchemy)을 사용하거나 단일 DB 엔진으로 전면 통일해야 합니다.

※ 아래 문서 내용 중 "모듈 분리"로 표기된 항목들은 원본의 "컨테이너 분리"에서 정정된 것입니다.

GC Safety Lane & Auditing Module 분리

가비지 컬렉션(GC) 작업은 비가역적인 데이터 하드 삭제와 RetiredIndexBridge 외부 API 연동이 밀접하게 결합되어 있어, 오작동 시 전체 원장(Monolithic Ledger) 데이터베이스에 미치는 블래스트 래디우스(Blast Radius)가 가장 큽니다. 따라서 GC 감사 로깅(memory_gc_audit) 및 백업 프로세스를 독립된 GC Safety Lane & Auditing Module로 격리하는 것을 최우선으로 권장합니다.

ledger.py 내부 책임 지도 (ledger.py Responsibility Map)

ledger.py는 SQLite 영속 데이터베이스를 활용하여 neurons 서버가 소유한 지식 아이템(Knowledge Items), 트랜스크립트 세션 및 턴, GC 감사 기록, 네이티브 메모리 미러링 등 전체 원장의 상태와 트랜잭션을 관리합니다. 내부 책임은 크게 다음 4가지 영역으로 나뉩니다.

As-built (2026-06-16): 이 4-area 경계는 audit override(Modular Monolith)대로 물리 분리 대신 in-process 경계로 구현됨 — worker/lib/agent_knowledge/ledger_areas.py(34테이블→4영역+core manifest) + worker/eval/ledger_area_boundaries.py lint(전수·배타 + cross-area 메서드 frozen allowlist). 정본은 spec docs/superpowers/specs/2026-06-15-ledger-refactor-autopilot-design.md §0.

아키텍처 개선 효과 (Wins)

  • Locality: GC 스케줄러 실행 및 감사 로그, 삭제 전 백업 로직이 독립적인 모듈에 격리되어 모놀리식 원장의 blast radius가 제한됩니다.
  • Leverage: RetiredIndexBridge 삭제 API의 일시 장애 대처 능력 및 재시도 메커니즘을 원장 트랜잭션과 독립적으로 구축하여 결함 감내가 향상됩니다.
  • Interface/Seam: SQLite 직접 쿼리 대신 REST/gRPC API Seam을 두어 결합도를 낮추고 결함 전파를 차단합니다.
  • Tests: 원장의 실제 DB 잠금 경합을 방지하고 GC 삭제 과정의 dry-run 및 validation을 독립적인 테스트 스위트로 집중 커버 가능합니다.

내부 모듈 분리(Modular Monolith) 비용 (Separation Cost)

Medium

RetiredIndexBridge API 및 로컬/원격 백업 디렉토리 경로에 대한 결합이 있고, 원장의 마킹 처리를 위해 Ledger Core API와의 네트워크 통신 흐름이 신설되어야 하기 때문입니다.

위험성 및 ADR 갈등 요소 (Risks)

GC 삭제 상태 기록과 원장의 활성 상태 간 일시적 불일치(Eventual Consistency)가 발생할 수 있어, API 타임아웃 및 정족수 검증 등의 재시도 로직이 추가적으로 요구됩니다.

Interface & Adapter Design

GC 스케줄러와 물리 삭제 흐름에서 결함을 격리하고 비가역성 문제를 감내하기 위해 감사 로그 저장을 전담하는 추상 인터페이스를 설계합니다. 

class IGCSafetyAuditor(ABC):
                @abstractmethod
                def log_gc_start(self, gc_kind: str, knowledge_id: str, document_id: str) -> str:
                    """GC 감사 시작 로깅 및 고유 audit_id 발행"""
                    pass

                @abstractmethod
                def log_gc_success(self, audit_id: str, replacement_id: str = "") -> None:
                    """물리 삭제 성공 시 감사 로그에 완료 기록"""
                    pass

                @abstractmethod
                def log_gc_failure(self, audit_id: str, error_msg: str) -> None:
                    """에러 로깅 및 결함 상태 복구 트리거"""
                    pass
Test Survival Strategy

단위 테스트 시 MockGCSafetyAuditor를 주입하여 실제 RetiredIndexBridge API의 HTTP 통신을 우회하고 감사 상태 전이 일관성만 고속으로 검증합니다.
